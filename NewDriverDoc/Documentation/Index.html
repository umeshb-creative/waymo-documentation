Waymo Automation Tool Architecture and Design Document

  1. Introduction

    1. Overview of the Waymo Automation Tool

      The Waymo Automation Tool is a robust and extensible cross-platform system designed to enable seamless integration, configuration, control, and monitoring of hardware devices used in automated manufacturing and testing environments. Developed with a modern microservice architecture, Waymo facilitates smooth communication between software components, embedded drivers, and a unified user interface, making it a powerful solution for industrial automation.

      At its core, Waymo is built on .NET Core 8.0, ensuring cross-platform compatibility and high-performance execution across Windows 10 and Windows 11 systems. It uses gRPC for client-server communication, which provides efficient binary serialization and supports multiple streaming options (unary, client streaming, server streaming, and bidirectional streaming), making it significantly faster and more secure than traditional HTTP REST APIs.

    2. Purpose of the Document

      This document provides a comprehensive and detailed description of the architecture and design of the Waymo Automation Tool Solution. It covers the hardware components, software modules, and their interactions, including the Human-Machine Interface (HMI) layer.

      It also outlines the rationale behind key design decisions, such as the choice of:

      • Operating systems
      • Hardware platforms
      • Communication interfaces
      • Software frameworks
      The goal is to present a holistic overview of the system and the reasoning behind its architectural choices.

  2. Problem Statement

    In the development and testing environment at Waymo, a wide variety of hardware devices are involved. These devices pose a challenge due to:

    • Diverse driver compatibility:
      • Some hardware supports only Linux drivers
      • Others are compatible only with Windows drivers
    The central challenge is to establish seamless communication and control over all these heterogeneous devices via a single, unified HMI application.

  3. HMI Application Requirements

    1. Drag-and-Drop Interface for Test Case Creation

      A user-friendly HMI enables users to visually create test cases by dragging and dropping devices

      • The HMI offers a visual interface that enables users to:
        • Define input-output relationships between components
        • Drag and drop devices and controls
      This simplifies test case creation and allows even non-technical users to configure hardware interaction logic easily.

    2. User Privileges and Access Control

      The system implements multi-level user privileges, offering role-based access for Administrators, Technicians, Engineers, and Operatorseach with defined permissions for creating, executing, or reviewing use cases and reports.

      • The application supports multiple user roles such as:
        • Administrator
        • Technician
        • Engineer
        • Operator
      • Each role has specific privileges, including:
        • Creating/modifying use cases
        • Executing use cases
        • Viewing logs and reports
      This ensures both flexibility and security in a collaborative environment.

    3. Test Case Execution and Reporting

      Users can execute defined test cases directly from the interface, with real-time monitoring and automated generation of detailed logs and reports for traceability, validation, and continuous improvement.

      • Users can:
        • Execute test cases with a single command
        • Monitor performance and behavior
        • Generate detailed reports for each session
      These reports help track test success/failure and evaluate hardware performance across different conditions.

    9. Security & Encryption

      GRPC uses HTTP/2 as its transport layer protocol to communicate. There are actually two types of checks happening on the client side: certificate check and hostname verification check. The former checks the server certificate to make sure the client trusts it; the latter checks the target name with the server's identity on the peer certificate. The server uses Https redirection where the server validates the certificate sent by client and then redirects the request. gRPC runs over HTTP/2 with TLS encryption, including:

      • Server certificate validation
      • Hostname verification
      • HTTPS redirection
      These ensure secure and trusted client-server interactions

Emergency Stop and Recovery Process

  1. Purpose

    This document outlines the standard procedures for initiating an emergency stop (E-Stop) and safely recovering tool or system operations after an emergency halt. It is intended to protect operators, equipment, and surrounding environments from harm or damage.

  2. Emergency Stop Procedure

    • When to Trigger Emergency Stop

      • Initiate an Emergency Stop immediately in the following cases:

      • Risk of injury to personnel
      • Sudden, unintended tool movement
      • Abnormal sound, vibration, or smoke
      • Malfunctioning safety interlocks
      • Electrical short circuit or sparks
      • Fire, fluid leakage, or hazardous gas release

    • How to Activate

      1. Locate the Emergency Stop Button or Switch
        • Clearly marked in red with a yellow background
        • Typically mushroom-style pushbutton
      2. Press the Emergency Stop Button Firmly
        • Tool/system should immediately cease operation
        • Power may be cut depending on system design
      3. Evacuate if Necessary
        • If risk to health or safety is present, follow site evacuation procedures
      4. Report the Incident
        • Inform supervisor or safety officer immediately
        • Fill out the Incident/Stop Report Form (see Section 6)

  3. Recovery ProcedureAuthorized Personnel Only

    Only trained and authorized technicians may perform recovery procedures.

    1. Perform Safety Assessment
      • Inspect the tool/system for visible damage or hazards
      • Ensure the cause of the stop is understood and resolved
    2. Reset Emergency Stop Button
      • Turn or pull the E-Stop button (as designed) to release
    3. Restore Power/System State
      • Re-enable system power if disconnected
      • Restart relevant subsystems (motors, sensors, network)
    4. Run Diagnostics or System Check
      • Use control software or onboard diagnostics
      • Check logs or alerts for error codes
    5. Test in Safe Mode
      • Operate tool/system in a restricted or manual mode to verify safety
    6. Resume Normal Operation
      • Only after all checks pass and supervisor gives approval

Introduction to Node-RED

  1. Introduction

    Node-RED is a flow-based development tool for wiring together hardware devices, APIs, and online services. Node-RED provides an easy-to-use visual interface for creating applications that interact with IoT devices, APIs, and web services. It is designed to simplify integration between various hardware and software, making it a popular choice for IoT, automation, and data-flow applications.

    1. What is Node-RED?

      Node-RED is an open-source, browser-based flow editor that enables users to connect devices, APIs, and online services in a seamless and visual manner. It uses a flow-based programming model where logic is constructed by wiring nodes together in a flow chart-like design. These nodes can represent hardware devices, API endpoints, databases, or even custom functions.

      Node-RED's easy-to-use interface and wide range of supported nodes make it an excellent choice for both beginners and advanced users to quickly create powerful applications without extensive programming knowledge.

    2. Key Features of Node-RED

      • Flow-based programming:The primary concept behind Node-RED is the flow-based programming paradigm. Users can create applications by connecting pre-configured nodes (blocks of code) to form logical sequences.
      • Drag-and-drop Interface:With its intuitive, web-based flow editor, Node-RED allows users to drag and drop nodes to quickly build applications.
      • Extensibility:Node-RED has a rich ecosystem of nodes available through the Node-RED library. You can add nodes for different communication protocols (HTTP, MQTT, Web-Sockets, etc.), data storage (SQL, NoSQL), and even machine learning (TensorFlow, Watson).
      • Integration:Node-RED makes it easy to integrate with IoT devices, third-party services, and cloud platforms such as AWS, IBM Watson, and Google Cloud. It supports a variety of communication protocols like MQTT, HTTP, Web-Sockets, and more.
      • Real-time monitoring and debugging:You can monitor, debug, and log data flows in real-time, making troubleshooting easier during development.
      • Custom Functions:For advanced users, Node-RED allows you to write custom JavaScript functions, providing flexibility, code re-usability and used for more complex use cases.

    3. Components of Node-RED:

      Node-RED is composed of three main components:

      • Flow Editor:This is the user interface where you design and configure flows. You can drag nodes from the left panel, connect them, and deploy them to the runtime. The flow editor runs in a web browser.
      • Nodes:Nodes are the building blocks of your application in Node-RED. They represent the various components that you can interact with, such as devices, services, or custom functionality. Node-RED provides pre-built nodes for common tasks like HTTP requests, MQTT messages, and GPIO (General Purpose Input/Output) operations.
      • Example nodes include:
        • Inject Node:Used to generate messages manually or on a schedule.
        • HTTP Request Node:Used to make HTTP requests to external services.
        • Function Node:Allows you to write custom JavaScript to process data. Similarly we have multiple nodes in node-red dashboard with its variety of nodes which is used to develop UI dashboard also.
      • Runtime:Node-RED runs on a server (local or remote), processing the logic defined in the flow editor. The runtime is responsible for executing the flows, listening for incoming events, and interacting with the connected devices or services.

  2. Setting Up Node-RED

    1. Installation:

      • Node-RED is built on Node.js and can be easily installed using npm (Node.js package manager).
      • To install Node-RED globally, run the following command in your terminal:

        • Bash or terminal

        npm install -g --unsafe-perm node-red

    2. Starting Node-RED:

      • After installation, you can start Node-RED by running:

        • Bash or terminal

        node-red

      • This will start the Node-RED service and open the flow editor at http://localhost:1880 in your web browser.

    3. Accessing the Flow Editor:

      • After starting Node-RED, you can access the flow editor by navigating to http://localhost:1880 in your browser.
      • You can begin creating flows by dragging nodes from the left sidebar and connecting them to form your application.

  3. Creating Your First Flow:

    1. Inject Node:

      To start, drag an Inject Node onto the flow editor canvas. This node will trigger an event, such as sending a message.

    2. Debug Node:

      Drag a Debug Node to the canvas. This node will output the results of the flow to the debug window.

    3. Connecting Nodes:

      Connect the Inject Node to the Debug Node by clicking the small squares on the sides of the nodes and dragging a line to connect them.

    4. Deploy the Flow:

Node-RED Execution and Flow Troubleshooting Guide

  1. Introduction

    Node-RED is a flow-based development tool where each node performs a specific task and passes the message (msg) to the next node. Below is a breakdown of flow behavior and lifecycle.

    1. Message Flow:

      • A flow begins with an input node (e.g., HTTP In, MQTT In, Inject).
      • The message flows through processing nodes (e.g., Function, Change, Switch).
      • It ends in output nodes (e.g., HTTP Response, MQTT Out, Debug).

    2. Execution Lifecycle:

      • Trigger:A flow is triggered by an event (e.g., Inject button, HTTP request).
      • Propagation:The msg object is passed from one node to the next via wires.
      • Processing:Each node processes the msg according to its logic.
      • Termination:The flow ends with a node that does not pass data further (like HTTP Response).

    3. Execution Context:

      Node-RED provides three types of data context:

      • msg: Per-message scope. Changes only apply to the current message.
      • flow: Shared across nodes within the same flow.
      • global: Accessible across all flows in the instance.

  2. Common Issues and How to Resolve Them

    1. Issue 1: Node Not Executing:

      Cause: Node conditions not met or previous node not passing data

      How To Fix

      • Use Debug nodes after each node to see if msg is passed.
      • Check wires to ensure nodes are connected properly.

    2. Issue 2: Undefined or Empty msg Properties

      Cause: Accessing properties that do not exist.

      How To Fix

      • Add checks in Function nodes:

        • if (!msg.payload) {
        node.error("Payload is missing", msg);
        return null;
        }

    3. Issue 3: Function Node Not Working

      Cause: Syntax errors or logic issues.

      How To Fix:

      • Check Node-RED logs for errors.
      • Use node.warn() and node.log() to trace.

    4. Issue 4: Inject Node Doesnt Trigger

      How To Fix:

      • Make sure node is deployed
      • Try manually clicking the Inject button.

    5. Issue 5: Flows Not Deploying

      How To Fix:

      • Ensure there are no syntax errors in Function nodes.
      • Check if any required nodes are missing (e.g., custom nodes).

  3. Advanced Debugging Techniques

    1. Use the Debug Panel

      • Set Debug nodes to show the complete message.
      • Place Debug nodes strategically across flows.

    2. Logging in Function Nodes

      node.warn("This is a warning");
      node.log("This is a log message");

    3. Catch and Status Nodes

      • Use Catch node to capture and log errors.
      • Use Status node to monitor node status (e.g., connected, error).

  4. Flow Management

    1. Flow Tabs

      • Organize logic across multiple flows.
      • Use Link In/Out nodes to manage flow transitions.

    2. Backups

      • Always keep exported JSON copies of important flows.
      • Use version control (Git) for production flows.

    3. Flow Configuration

      • Use Environment Variables for dynamic values.
      • Use Context Storage to persist data across deploys.
      • Leverage Credentials storage for secrets like API keys.

  5. Node-RED Runtime Troubleshooting

    1. Understanding Runtime Architecture

      • Node-RED runs on Node.js.
      • Each flow is compiled to a set of JavaScript functions.
      • Nodes run sequentially, and asynchronously if needed.
      • Runtime maintains a single-threaded event loop but leverages async I/O.

    2. Accessing Logs

      • Use terminal:

        node-red-log

      • Or check logs in

        /var/log/node-red.log

    3. Restart Node-RED

      • Run Command in terminal

        sudo systemctl restart nodered

      • Checking Port Usage

        netstat -tulnp | grep :1880

    4. Performance Monitoring

      • Use top, htop, or Node-REDs built-in metrics (if enabled).
      • Consider enabling Node-RED Flow Metrics to track throughput and delays.

  6. Debugging Custom Nodes

    • Ensure the node is installed properly using npm.
    • Restart Node-RED after installing custom nodes.
    • Check the nodes source code in ~/.node-red/node_modules/
    • Use try/catch inside the custom node function for error handling.

  7. Troubleshooting Flows

    1. Tips for Diagnosing Flow Issues

      • Use Debug Nodes generously to trace the path of msg.
      • Validate logic in Switch and Change nodes.
      • Inspect Function nodes for proper output.
      • Review flow triggers some flows may not trigger without specific conditions.

    2. Ensuring Flow Reliability

      • Avoid hard-coding paths and values.
      • Use environment variables or configuration nodes.
      • Group common patterns into subflows for reusability.

    3. Preventing Flow Errors

      • Always check if the required properties exist in msg.
      • Wrap logic in try/catch blocks if using advanced scripting.
      • Use return null; to stop flows when needed.

    4. Handling External APIs or Inputs

      • Add Retry/Timeout mechanisms.
      • Use Status and Catch nodes to handle unexpected failures
