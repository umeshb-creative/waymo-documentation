Denkovi Driver Document

Introduction

Brief overview of the project

This project focuses on developing drivers for Denkovi to be utilized in automation applications, with a particular emphasis on integrating them with a user interface (UI) application for future use. The project aims to create software components that enable seamless communication between denkovi and the UI application, allowing for intuitive control and monitoring of the IO systems. Through this integration, the project seeks to enhance the usability and functionality of Denkovi Relay within automated processes, contributing to increased efficiency and productivity in various industrial settings

Purpose of the document

The purpose of this document is to provide a detailed guide for developing a driver for the Denkovi Relay device. The driver will enable communication between the Relay channel and external software applications. It covers the software development process, including setting up the development environment, implementing the driver, testing and debugging, and integrating the driver with the denkovi device.

The primary objective of the document is to provide a comprehensive guide for developed drivers for Denkovi used in automation using a UI application. This includes detailed instructions, code examples, and best practices to ensure the successful development and integration of drivers with the devices.

Device Overview

Description of the device for which the driver is being developed

Five channel Ethernet relay card based on DAEnetIP4 snmp IP controller. Additionaly it supports 8 digital outputs, 8 digital inputs, 8 analog inputs plus 2 PWM outputs. Control it from web browser,command line tool, free Android and iOS App and others. Flexible for integration.

Some of the applications of the Denkovi Relay includes:
  • Monitoring and Logging
  • Home automation
  • Security and fire alarm system
  • Remote lock and unlock doors
  • Irrigation system

Driver Development

Introduction to driver development for the device

The driver development consists of three key components:

  • denkovi.py

    • denkovi.py serves as the core interface, offering SNMP protocol oid’s that encapsulate the minimum functionalities required by denkovi devices, as specified by Waymo. This code provides the all scratch code for the required API.

  • Denkovi_driver.py

    • Denkovi_driver.py plays a crucial role in establishing communication between the software and Denkovi devices. It provides APIs to connect to the device, send and receive packets, and encode/decode data according to the Denkovi protocol. This code enables the minimum functionalities of the API.

Choice of Programing Language

For developing the driver we have used Python. Python3 will support for this denkovi.

gRPC API’s Description for Denkovi Device

  1. GetDeviceDetails

    Description for GetDeviceDetails API: To Connect to the Denkovi Device and Capture Device Information.

    RPC Command : rpc GetDeviceDetails(.generic. CommandRequest) returns (.generic.GenericDeviceModel) {}

    Request:
    message CommandRequest {
    string device_alias = 1;
    }
    Parameter Description
    device_alias Unique alias of the device
    Request:
    message GenericDeviceModel{
    string instance_id = 1;
    string device_alias = 2;
    DeviceType device_type = 3;
    DeviceModel device_model_name = 4;
    optional DeviceStatus device_status = 5;
    string ipaddr = 6;
    string mac_addr = 7;
    google.protobuf.Value conn_handle=8;
    string port_name = 9;
    int64 port_address = 10;
    CommandResponse cmd_response = 11;
    }
    Parameter Description
    instance_id New instance for the device in the enumeration
    device_alias Unique identifier (alphanumeric) for every device in the enumeration
    device_type Indicates the type of device like Motion stage, Safety Controller etc (defined below in the DeviceType enum)
    device_model_name Indicate the model name for each device (define below in the DeviceModel enum)
    device_status Indicate whether the device is online , offline or not reachable (define below in the device status enum)
    ipaddr indicate the ipaddress of the device
    mac_addr Indicate the mac address of the device
    Conn_handle Handle object to keep the device open
    port_name Port name of the device
    port_address Port address of the device
    cmd_response Indicates command response that describes API response code and its description (Defined below in CommandResponse)
    enum DeviceType{
    NONE = 0;
    SENSOR_3D = 1;
    MOTION_STAGE = 2;
    CAMERA = 3;
    IO_MODULE = 4;
    SAFETY_CONTROLLER = 5;
    ANIMATIC_MOTOR=6;
    }
    enum DeviceModel{
    DUMMY_MODEL = 0;
    ED527 = 1;
    ED588 = 2 ;
    JR3000 = 3;
    OPTOE = 4;
    BASLER = 5;
    SPiiPlusEC=6;
    GT6600 = 7;
    MOTOR = 8;
    }
    enum DeviceStatus{
    OFFLINE = 0;
    ONLINE = 1;
    NOT_REACHABLE =2;
    }
    message CommandResponse{
    int32 resp_code = 1;
    string resp_message = 2;
    }
    Parameter Description
    resp_code Unique alias of the device
    resp_message Description of the response code

    CLI Command: python denkovi.py GetDeviceDetails DENKOVI_1

    Description for CLI Command:
    • GetDeviceDetails: GetDeviceDetails is the name of the API
    • DENKOVI_1: Aliased Name for Denkovi

    • Ensure to giving the valid channel number and state.

GLOSSARY

  • Denkovi Driver: Software module or library responsible for interfacing with denkovi relay module.
  • API (Application Programming Interface): Set of protocols, tools, and definitions that allow software applications to communicate and interact with the Denkovi driver.
  • Command: Instruction or directive sent to a Denkovi I/O through the driver to perform specific actions such as movement, I/O control, or program execution.
  • Response: Feedback or acknowledgment received from the Denkovi I/O after executing a command, indicating the success or failure of the operation and providing relevant information or data.
  • I/O (Input/Output):: Interface for interacting with external devices or sensors connected to the Denkovi relay module, allowing the driver to control outputs.
  • OID OID expansion is Object Identifier.The OID is used to refer to unique characteristics and navigate through variables on the connected device. The value of these identifiers varies from text to numbers and counters.
  • SNMP: SNMP or Simple Network Management Protocol is a well-known networking protocol that sits at the application layer. SNMP is used to monitor SNMP-enabled devices and view how their performance is holding up. SNMP architecture is made up of SNMP managers and SNMP agents.
