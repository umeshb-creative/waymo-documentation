Brain Box driver document

Introduction

Brief overview of the project

This project focuses on developing drivers for Brainbox ED588 and ED527 to be utilized in automation applications, with a particular emphasis on integrating them with a user interface (UI) application for future use. The project aims to create software components that enable seamless communication between brainbox and the UI application, allowing for intuitive control and monitoring of the IO systems. Through this integration, the project seeks to enhance the usability and functionality of Brainbox Relay within automated processes, contributing to increased efficiency and productivity in various industrial settings.

Purpose of the document

The purpose of this document is to provide a detailed guide for developing a driver for the brainbox ED588 and ED527 device. The driver will enable communication between the IO module and external software applications. It covers the software development process, including setting up the development environment, implementing the driver, testing and debugging, and integrating the driver with the brainbox device.

The primary objective of the document is to provide a comprehensive guide for developed drivers for brainbox used in automation using a UI application. This includes detailed instructions, code examples, and best practices to ensure the successful development and integration of drivers with the devices.

Scope and objectives

  1. Driver Development: The document focuses on the development of a driver for brainbox device.
  2. Technical Information: It includes technical specifications of the device, hardware and software requirements, and information about the Software Development Kit (SDK) provided for the device.
  3. SDK Overview: Introduction to Boost.IO manager and its Installation.
  4. Driver Implementation: It covers the development process, including the choice of programming language, information on APIs, how to use APIs, limitations of the driver, precautions to take before execution, and the actual implementation of the driver.
  5. Testing and Debugging: The scope extends to testing and debugging procedures to ensure the reliability and functionality of the driver.
  6. Documentation and Support: The document serves as a comprehensive resource for developers, offering detailed information on the BrainBox, Software Installation Links required for software and drivers, and reference videos for setup, configuration etc

Overall, the aim is to facilitate the successful development and implementation of the Brainbox driver, ensuring it meets the required standards and serves its intended purpose effectively.

Device Overview

Description of the device for which the driver is being developed

The Brainboxes Ethernet to digital I/O media converter has 8 non-isolated digital input channels and 8 digital output channels. It offers a robust and intelligent ’control side’ devices that provide the interface to systems on the 24VDC factory floor environment, commonly referred to as ’process side’.

Some of the Features of the brainbox IO includes:

  • Control any input or output from TTL to high voltage.
  • High current industrial device
  • Integrated DIN rail mount
  • RS485 gateway port

SDK OVERVIEW

The Boost, IO Manager is a graphical user interface application which will allow you to find your Brainboxes ED device on a network and then configure some of the settings in Windows. As part of the Boost.IO application a COM port can be installed which allows communication with the device using the ASCII protocol. Boost.IO Manager is not required if you want to communicate with the device using just the IP address or if you want to configure the device only using the web configuration pages.

gRPC API’s Description for Brainbox

  1. OpenDevice

    Description for OpenDevice API: Establishes a connection to the Brain Box, initializing communication for further operations.

    RPC Command : rpc OpenDevice(.generic.CommandIPRequest) returns(.generic.GenericDeviceModel) {}
    Request:
    message CommandIPRequest{
    string ip_address = 1;
    string com=2;
    int64 port_address=3;
    int32 device_id = 4;
    string file_path = 5;
    int64 device_model =6;
    optional string device_alias=7;
    int32 axis =8;
    }
    Parameter Description
    ip_address ip_address IP Address of the Device
    com COM Port Address of the Device with serial communication(ex. COM1)
    port_address Port address of the device with serial communication (ex. 1)
    Response
    message GenericDeviceModel{
    string instance_id = 1;
    string device_alias = 2;
    DeviceType device_type = 3;
    DeviceModel device_model_name = 4;
    optional DeviceStatus device_status = 5;
    string ipaddr = 6;
    string mac_addr = 7;
    google.protobuf.Value conn_handle=8;
    string port_name = 9;
    int64 port_address = 10;
    CommandResponse cmd_response = 11;
    }
    Parameter Description
    instance_id New instance for the device in the enumeration
    device_alias Unique identifier (alphanumeric) for every device in the enumeration
    device_type Indicates the type of device like Motion stage, Safety Controller etc (defined below in the DeviceType enum)
    device_model_name Indicate the model name for each device (define below in the DeviceModel enum)
    device_status Indicate whether the device is online , offline or not reachable (define below in the device status enum)
    ipaddr Indicate the ipaddress of the device
    mac_addr Indicate the mac address of the device
    Conn_handle Handle object to keep the device open
    port_name Port name
    port_address Port address
    cmd_response Indicates command response that describes API response code and its description (Defined below in CommandResponse)

    enum DeviceType{
    NONE = 0;
    SENSOR_3D = 1;
    MOTION_STAGE = 2;
    CAMERA = 3;
    IO_MODULE = 4;
    SAFETY_CONTROLLER = 5;
    ANIMATIC_MOTOR=6;
    }
    enum DeviceModel{
    DUMMY_MODEL = 0;
    ED527 = 1;
    ED588 = 2 ;
    JR3000 = 3;
    OPTOE = 4;
    BASLER = 5;
    SPiiPlusEC=6;
    GT6600 = 7;
    MOTOR = 8;
    }
    enum DeviceStatus{
    OFFLINE = 0;
    ONLINE = 1;
    NOT_REACHABLE =2;
    }
    message CommandResponse{
    int32 resp_code = 1;
    string resp_message = 2;
    }
    ParameterDescription
    resp_code Response code from the device API
    resp_message Description of the response code

    CLI Command: python brainbox.py OpenDevice 192.168.127.112 BRAINBOX_2

    Description for CLI Command
    • OpenDevice: API called to the Brain Box through CLI Command to connect and initialize all resources
    • 192.168.127.112: IP Address of the Device
    • BRAINBOX_2: Aliased Name for Brainbox

  5. SetIO

    Description of the API: API sets or resets a specific output pin on a device. This allows for controlling an output pin's state (either High or Low) based on the provided parameters.
    RPC Command: rpc SetIO (BrainboxWriteIORequest) returns (CommandResponse) {}
    Request: message BrainboxWriteIORequest {
    int32 io_num = 1;
    bool io_state = 2;
    .generic.CommandRequest cmd_request = 3;
    }
     message CommandRequest {
    string device_alias = 1;
    }
    Parameter Description
    io_num For the ED588, output pins are numbered from 0 to 7, allowing you to set or reset any of these 8 pins.For the ED527, output pins are numbered from 0 to 15, providing the ability to control any of these 16 pins.
    Io_state The status be to configured. 0 represents RESET and 1 represents SET.
    device_alias The unique device_alias used to refer the device
    Response: message CommandResponse {
    int32 resp_code = 1;
    string resp_description = 2;
    }
    Parameter Description
    resp_code Response code from the device API
    resp_description Description of the response code

    CLI Command: python brainbox.py SetIO BRAINBOX_2 4 0

    Description for CLI Command:
    • SetIO: API called to the Brain Box through CLI Command
    • BRAINBOX_2: Aliased name of the Device
    • 4: Specific output pin number of Brainbox device to be Set or Reset.
    • 0: Setting the pin as Low(Reset)

Limitations of the driver

Compatibility: The driver only supports the ED-588 and ED-527 Module not work with other module.

Integration: Integrating the driver with other systems or software may pose challenges, especially if there are compatibility issues or dependencies.

Precautions to take before executing the driver

  • Make sure the Device is in connected with the station
  • Ensure to giving the valid channel number and state.

GLOSSARY

When developing a Brainbox driver, it's helpful to have a glossary of terms to ensure clarity and consistency in communication. Here's a glossary of terms commonly used in Brainbox driver development:

  • Brainbox Driver: Software module or library responsible for interfacing with Brainbox module.
  • API (Application Programming Interface): Set of protocols, tools, and definitions that allow software applications to communicate and interact with the Brainbox driver.
  • Command: Instruction or directive sent to a Brainbox robot through the driver to perform specific actions such as movement, I/O control, or program execution.
  • Response: Feedback or acknowledgment received from the Brainbox after executing a command, indicating the success or failure of the operation and providing relevant information or data.
  • I/O (Input/Output): Interface for interacting with external devices or sensors connected to the Brainbox relay module, allowing the driver to control outputs.
  • ASCII: ASCII TCP is a connection-based protocol, in which the client establishes a connection with the server and the server waits for an incoming connection from the client. Once a connection is established, the ASCII command set is used to configure devices, send data to devices and to read data and status information back from devices. This occurs until the client closes the connection.
