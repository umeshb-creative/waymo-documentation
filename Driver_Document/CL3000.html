Keyence CL3000 driver document

Introduction

Brief overview of the Project

The CL3000 driver project is focused on developing a robust software interface that ensures efficient communication between the CL3000 confocal displacement sensor and external systems. This driver enables precise control, data acquisition, and management of the sensor, ensuring accurate and reliable measurements for diverse applications, including surface profiling, thickness measurement, and material inspection. The project's goal is to streamline sensor integration and optimize its performance in industrial and research environments.

Purpose of the document

The purpose of this document is to provide a detailed guide for developing a driver for the CL3000 device. The driver will enable communication between the device and external software applications. It covers the software development process, including setting up the development environment, implementing the driver, testing and debugging, and integrating the driver with the CL3000 device.

The primary objective of the document is to provide a comprehensive guide for developed drivers for CL3000. This includes detailed instructions, code examples, and best practices to ensure the successful development and integration of drivers with the devices.

Scope and objectives

  1. Technical Information: Document provides comprehensive technical details about the CL3000 sensor, including its features, specifications, and operational principles, offering users a thorough understanding of its capabilities and functionalities.
  2. SDK Overview: The document includes an overview of the software development kit (SDK) associated with the CL3000 device, offering developers insights into integrating the sensor into their applications and leveraging its capabilities effectively.
  3. Driver Development: It outlines the development process for the CL3000 driver, detailing the steps involved in creating software that enables communication between the sensor and the operating system. This section includes information on APIs, libraries, and programming languages used for driver development.
  4. Driver Implementation: The document contains guidelines and best practices for implementing the CL3000 driver, ensuring compatibility with various operating systems and hardware configurations.
  5. Testing and Debugging: This section covers strategies and methodologies for testing the CL3000 driver, ensuring its reliability, stability, and performance under different conditions. It also provides guidance on debugging techniques to troubleshoot any issues that may arise during driver development or deployment.
  6. Documentation and Support: The document serves as a comprehensive resource for developers, offering detailed documentation on the CL3000 device, driver, and SDK.

Device Overview

Description of the device for which the driver is being developed

The CL3000 device is a confocal displacement sensor designed for high-precision measurements on various materials and surfaces. Key features and characteristics of the CL3000 device include:

  • Ultra-compact coaxial laser displacement sensor suitable for diverse applications.
  • Superior precision and accuracy, unaffected by external factors such as heat or electrical noise.
  • Multi-color confocal method for enhanced performance and stability.
  • Easy installation and integration, even in constrained spaces or on robotic platforms.
  • Effective on curved, uneven, and rough surfaces, ensuring reliable measurement across different target shapes and finishes.
  • Compatibility with transparent, mirrored, metallic, ceramic, and adhesive surfaces.
  • Lightweight design for easy integration with robotic systems, reducing vibration and equipment costs.
  • Utilization of ultra-bright light sources for stable measurements over the sensor's full range.

Overall, the CL3000 device represents a cutting-edge solution for manufacturers seeking precise, reliable, and versatile measurement capabilities for quality control and production optimization.

SDK OVERVIEW

The CL3000 Interface Software Suite contains a powerful SDK and helps to develop CL3000 application for Windows on desktop as well as on embedded systems.

With the well-designed and user-friendly API, we will need just a few lines of code to configure the run the device as it supports many languages like C++, CS, .NET languages.

File Structure and Usage the SDK(CL-NavigatorN_1_6_0_0)

The files required at execution are listed below.

CL3_IF.dll : DLL defines the interface for controlling user applications and the CL-3000 Series. CL control can be used to control the system, control the measurements, and modify/update settings. This DLL can also be used to control up to three CLs with one PC application. (However, only one CL can be connected at a time for USB communication.) When multiple CLs are connected, specify a value from 0 to 2 for deviceId when connecting over an Ethernet. Assuming that deviceId is set to 0 for IP address 192.168.0.1 and to 1 for IP address 192.168.0.2, specifying 0 for deviceId(in) when each command is sent results in these commands being sent to the controller with IP address 192.168.0.1.

Linking : The library can be linked implicitly or explicitly. To implicitly link the library, link with “CL3_IF.lib”. “CL3_IF.lib” was built with Visual C++ 2013.

Include files : Include the following header files in the necessary source files.

  • CL3_IF.h
  • CL3_ErrorCode.h

Device Interface

  • USB for PC application

    This controller can issue control instructions from multiple interfaces (I/O terminals, RS-232C, connection to CL-NavigatorN via USB or LAN, and EtherNet/IP communications) simultaneously. However, it cannot receive any instructions from other interfaces while receiving a control instruction from an interface.

  • Ethernet / Serial / PLC-Link / EtherNet/ Profinet / EtherCAT for command control

    The Ethernet port on this controller can be used to communicate with external equipment in addition to connecting to our PC application software (CL-NavigatorN). To communicate with external devices, you can use the non-procedural mode (which has the same commands as RS-232C), EtherNet/IP, PLC-Link, PROFINET, or EtherCAT. The IP address, port number, subnet mask, gateway, etc. in the system settings of the controller will be used.

Developers working with the CL-Navigator SDK who wish to create their own C++ applications but do not want to invest in a commercial Integrated Development Environment (IDE) can leverage free Microsoft Visual Studio tools. These tools provide a cost-effective alternative for building C++ applications while supporting .NET languages like C# and Visual Basic, both of which are compatible with the CL-Navigator SDK.

In the past, Microsoft offered single-language programming environments, but these are no longer available. Instead, users must install the full edition of free Microsoft Visual Studio, which provides all the necessary tools for building C++ applications alongside .NET applications. Below are two recommended free versions of Microsoft Visual Studio for developing C++ applications based on the CL-Navigator SDK:

  • Microsoft Visual Studio Express 2013 for Windows Desktop: Although this version does not include the x64 native compiler, it supports the creation of 64-bit executables using the x86-x64 cross compiler running on WOW64.
  • Microsoft Visual Studio Community: Serving as the successor to the Express Edition starting with the 2015 release, Microsoft Visual Studio Community is also available in a 2013 release. It offers functionality comparable to the full Professional Edition of Visual Studio.

  1. OpenDevice

    Description for OpenDevice API: Establishes a connection to the CL3000, initializing communication for further operations.

    RPC Command : rpc OpenDevice(.generic.CommandIPRequest) returns(.generic.GenericDeviceModel) {}
    Request:
    message CommandIPRequest{
    string ip_address = 1;
    string com=2;
    int64 port_address=3;
    int32 device_id = 4;
    string file_path = 5;
    int64 device_model =6;
    optional string device_alias=7;
    int32 axis =8;
    }
    Parameter Description
    ip_address ip_address IP Address of the Device
    com COM Port Address of the Device with serial communication(ex. COM1) Not applicable for CL3000
    port_address Port address of the device with serial communication (ex. 1)
    Response
    message GenericDeviceModel{
    string instance_id = 1;
    string device_alias = 2;
    DeviceType device_type = 3;
    DeviceModel device_model_name = 4;
    optional DeviceStatus device_status = 5;
    string ipaddr = 6;
    string mac_addr = 7;
    google.protobuf.Value conn_handle=8;
    string port_name = 9;
    int64 port_address = 10;
    CommandResponse cmd_response = 11;
    }
    Parameter Description
    instance_id New instance for the device in the enumeration
    device_alias Unique identifier (alphanumeric) for every device in the enumeration
    device_type Indicates the type of device like Motion stage, Safety Controller etc (defined below in the DeviceType enum)
    device_model_name Indicate the model name for each device (define below in the DeviceModel enum)
    device_status Indicate whether the device is online , offline or not reachable (define below in the device status enum)
    ipaddr Indicate the ipaddress of the device
    mac_addr Indicate the mac address of the device
    Conn_handle Handle object to keep the device open
    port_name Port name
    port_address Port address
    cmd_response Indicates command response that describes API response code and its description (Defined below in CommandResponse)

    enum DeviceType{
    NONE = 0;
    SENSOR_3D = 1;
    MOTION_STAGE = 2;
    CAMERA = 3;
    IO_MODULE = 4;
    SAFETY_CONTROLLER = 5;
    ANIMATIC_MOTOR=6;
    }
    enum DeviceModel{
    DUMMY_MODEL = 0;
    ED527 = 1;
    ED588 = 2 ;
    JR3000 = 3;
    OPTOE = 4;
    BASLER = 5;
    SPiiPlusEC=6;
    GT6600 = 7;
    MOTOR = 8;
    }
    enum DeviceStatus{
    OFFLINE = 0;
    ONLINE = 1;
    NOT_REACHABLE =2;
    }
    message CommandResponse{
    int32 resp_code = 1;
    string resp_message = 2;
    }
    ParameterDescription
    resp_code resp_code Response code from the device API
    resp_message resp_message Description of the response code

    CLI Command:python cl3000.py OpenDevice 192.168.1.20 CL3000_1

Limitations of the driver

While the CL3000 has impressive capabilities in high-precision measurement across various surfaces and materials, it's essential to acknowledge its limitations. Here are some potential constraints or limitations associated with the CL3000:

  1. Target Material Compatibility: While the CL3000 is designed to measure accurately on various surfaces, including transparent, mirrored, metallic, ceramic, and adhesive surfaces, there may be limitations in measuring certain types of materials or surfaces with extreme properties, such as highly reflective or absorptive materials.
  2. Measurement Range: Like any sensor, the CL3000 may have limitations in its measurement range. While it offers stable measurement over its full range, there may be constraints in terms of the minimum and maximum distances or displacements it can accurately measure.
  3. Environmental Factors: Although the CL3000 is designed to minimize the influence of external factors such as heat or electrical noise, extreme environmental conditions such as high temperatures, electromagnetic interference, or vibrations may still affect its performance.
  4. Complex Surfaces: While the CL3000 is effective on curved, uneven, and rough surfaces, there may be limitations in measuring extremely complex surfaces with intricate geometries or features.
  5. Integration Complexity: Integrating the CL3000 into specific applications or systems may require additional hardware, software, or customization, depending on the requirements of the application. Complex integration scenarios may pose challenges in terms of compatibility, calibration, or synchronization with other devices.
  6. Maintenance and Calibration: Like any precision instrument, the CL3000 may require regular maintenance and calibration to ensure accurate and reliable performance over time. Failure to perform proper maintenance or calibration may lead to degradation in measurement accuracy.

Precautions to take before executing the driver

  • Make sure to use the solution provided directly.
  • Make sure to add stdafx.h at the beginning to avoid cout, endl errors, std library errors.
  • We can’t build the solution at once. It will give below error as second project uses first project’s dll which is dependent in the build process. Try checking with individual project builds.
  • Make sure to set 64 bit and not 32 bit for C++ and C# projects to avoid the system exception issue as shown below.
  • Max Storage Index when buffer full is 100000.
  • Before storing data, there are a few settings to be configured in CL-Navigator. For Example, selecting OUT values to be stored. By default, no OUT is selected. So, storage won’t be started even you get success response. (Go to tools à Data storage in CL navigator for settings).
